# DIAGRAM_RULES.md

## 目的
本規範用於約束 Codex / LLM 在專案中自動產生各類圖表時的決策流程、文字表示方式、版面美學、統計圖表規則、輸出格式與文件整合規則。

核心目標如下：

1. 先判斷圖的類型
2. 自動切換到最適合的文字表示法
3. 自動套用統一的簡報美學規範
4. 優先輸出可編輯的 PPTX 圖形版面
5. 視需要同步輸出 PNG / SVG / Word 嵌圖版本
6. 確保 Word 內圖表可交互參考、順序正確、內文有引用
7. 圖片匯入 Word 前必須先裁掉四周多餘空白
8. 顏色必須具語義，相同顏色代表相同或相似群組
9. 預設以繁體中文排版，優先使用標楷體風格
10. 數值型資料優先使用統計圖表模板，不要硬塞進流程圖

---

## 一、總原則

### 1. 圖的生成分為四層
- **資料判斷層**：先判斷需求屬於流程、架構、拓樸、概念圖，還是數值統計圖表
- **結構描述層**：用文字式語言描述圖的邏輯結構或圖表定義
- **視覺排版層**：將結構轉為 PPTX 可編輯圖形、表格、文字方塊、圖示與箭頭
- **輸出層**：輸出 PNG / SVG / PPTX / Word 等成果檔

### 2. 簡報型用途優先
若用途是下列情境，最終輸出必須以 **PPTX 可編輯版** 為主：
- 計畫書
- 提案簡報
- 審查會議
- 教學簡報
- 說明會
- 對外展示
- 主管簡報

### 3. 不要只輸出圖片
每次產圖至少保留：
- 原始文字描述檔或資料定義檔
- PNG 或 SVG
- PPTX 可編輯版本

### 4. 採用 SmartArt 風格，但不強依賴 SmartArt 物件
PPTX 呈現優先使用：
- 文字方塊
- 圓角矩形
- 箭頭
- 線條
- 表格
- 色塊區域
- 小圖示
- 圖片
- 圖表元件

以模擬 SmartArt 的整齊感與層級感，避免過度依賴 PowerPoint 內建 SmartArt 本體。

### 5. Word 文件整合是正式輸出流程的一部分
若圖表要嵌入 Word，這不是額外動作，而是正式交付流程的一部分，必須在輸出過程中自動完成以下事項：
- 圖片先去除四周空白
- 依序插入圖表
- 自動建立圖說標題
- 內文建立交互參考
- 檢查圖號順序
- 確保所有圖表都被內文引用至少一次

### 6. 數值資料要優先判斷是否應改用統計圖表
若使用者提供的是：
- 百分比
- 比例
- 組別比較
- 時間序列
- 次數分布
- 前後差異
- 問卷統計
- KPI 趨勢
- 成本結構
- 市占結構

則應優先考慮使用統計圖表，而不是流程圖或概念框圖。

## 二、圖類型判斷規則

當使用者要求畫圖時，先依需求內容判斷圖種，再選擇對應的文字表示法與輸出策略。

### A. 流程圖 / SOP / 審查流程 / 作業流程
**適用關鍵詞：**
- 流程
- SOP
- 審查流程
- 作業流程
- 教學流程
- review flow
- concept flow
- PDCA
- 申請流程
- 核准流程

**結構描述語言：**
- Mermaid flowchart

**正式輸出：**
- PPTX 流程圖版面
- 視需要另輸出 PNG / SVG

**版面方向：**
- 主流程固定左到右
- 支援少量上下分支，但主視覺仍以左到右為主

---

### B. 系統架構圖 / 平台架構圖 / 模組圖
**適用關鍵詞：**
- 系統架構
- 平台架構
- 模組圖
- solution architecture
- technical architecture
- component
- deployment
- service architecture

**結構描述語言：**
- PlantUML component
- PlantUML deployment
- Mermaid architecture
- Mermaid flowchart（簡化版）

**正式輸出：**
- PPTX 模組架構圖
- 視需要另輸出 PNG / SVG

**版面方向：**
- 上下分層或左右分層皆可
- 需明確區分：前端 / 後端 / 資料層 / 邊緣設備 / 外部系統

---

### C. 網路拓樸圖 / 節點圖 / 伺服器關聯圖
**適用關鍵詞：**
- 網路拓樸
- 節點圖
- server topology
- network diagram
- cluster
- Ceph
- PVE
- VM
- LXC
- VPN
- gateway

**結構描述語言：**
- Graphviz DOT

**正式輸出：**
- PPTX 拓樸圖版面
- 技術文件可附 SVG / PNG

**版面方向：**
- 核心節點置中
- 周邊節點向外展開
- 線條不可過度交叉

### D. 組織架構圖 / 角色分工圖 / 階層樹狀圖
**適用關鍵詞：**
- 組織圖
- 架構圖
- 階層
- 職責分工
- 角色分工
- 樹狀圖

**結構描述語言：**
- Mermaid flowchart
- Mermaid mindmap
- 樹狀結構 JSON（若要轉 PPTX）

**正式輸出：**
- PPTX SmartArt 風格階層圖

**版面方向：**
- 自上而下
- 同層節點水平對齊

---

### E. 時序圖 / API 呼叫圖 / 系統互動圖
**適用關鍵詞：**
- 時序圖
- sequence
- API 呼叫
- 系統互動
- request/response
- 服務交握

**結構描述語言：**
- PlantUML sequence
- Mermaid sequenceDiagram

**正式輸出：**
- 技術文件：SVG / PNG
- 簡報：簡化成 PPTX 泳道圖

---

### F. 甘特圖 / 時程表 / 里程碑圖
**適用關鍵詞：**
- 甘特圖
- 時程
- timeline
- schedule
- milestone
- roadmap

**結構描述語言：**
- Mermaid gantt

**正式輸出：**
- PPTX 表格式時程圖
- 視需要輸出 PNG

---

### G. 提案圖 / 概念圖 / 簡報型比較圖 / 方案視覺圖
**適用關鍵詞：**
- 提案圖
- 概念圖
- 比較圖
- 商業模式圖
- 類產線示意圖
- 方案差異圖
- 特色總覽圖
- 價值鏈圖

**結構描述語言：**
- 不優先用工程圖語言
- 優先使用結構化 JSON 或 Python dict 描述版面區塊
- 再轉為 PPTX 文字方塊、圖示、表格、色塊組合

**正式輸出：**
- PPTX 可編輯版面
- 視需要另輸出 PNG

**版面方向：**
- 以簡報美觀為優先
- 不強求工程式連線語法

---

### H. 統計圖表 / 數據圖 / 問卷圖表 / KPI 視覺化
**適用關鍵詞：**
- 統計圖
- 圓餅圖
- 長條圖
- 柱狀圖
- 趨勢圖
- 折線圖
- 問卷分析
- 百分比
- KPI
- 數據比較
- 前後比較
- 分布圖
- 結構占比

**資料描述格式：**
- JSON / CSV / Python dict
- 欄位至少包含：
  - title
  - chart_type
  - categories
  - values
  - series
  - units
  - notes

**正式輸出：**
- PPTX 圖表
- PNG
- 若要進 Word，需附圖說與內文引用

**優先圖表類型判斷：**
- 比例結構 → 圓餅圖 / 環圈圖
- 類別比較 → 長條圖 / 群組長條圖
- 時間趨勢 → 折線圖
- 前後差異 → 雙柱狀圖 / 斜率圖
- 分布 / 排名 → 水平長條圖
- 累積結構 → 堆疊長條圖
- 問卷 Likert → 堆疊長條圖
- 散點關聯 → 散點圖

## 四、簡報美學硬性規範

以下規則為所有圖形與圖表預設必須遵守的視覺標準。

### 1. 標題規範
- 標題置頂且置中
- 標題與圖內容需保留固定留白
- 若有副標，副標置於標題下方，字級較小
- 標題不可過長，必要時拆為主標與副標

### 2. 節點規範
- 同層節點統一尺寸
- 每列不超過 4 個節點
- 同一類型節點必須同樣式
- 所有節點採固定圓角矩形
- 重要節點可略大，但不可破壞整體整齊度
- 每個節點文字不超過 12 字
- 每節點最多 2 行文字
- 文字過長時需拆為節點與註解框

### 3. 流向規範
- 主流程固定左到右
- 輔助流程使用虛線
- 箭頭方向一致
- 同一張圖內避免混合左右上下主要流向
- 不可出現太多交叉線
- 若分支過多，應拆成多張圖或分區塊

### 4. 配色規範
同圖最多只允許三類主要顏色：

- **主色**：深藍 / 藍綠
- **輔色**：橘 / 黃
- **中性色**：灰 / 淺灰

套用原則：
- 主流程與核心模組用主色
- 決策點、提醒、重點模組用輔色
- 註解、背景、次要元件用中性色

禁止：
- 超過 3 個主色
- 彩虹式配色
- 高飽和螢光色
- 同頁色彩邏輯混亂

### 5. 色彩語義規範
- 相同顏色必須代表相同或相似群組
- 不可讓同一顏色同時代表不同語意
- 群組顏色需在圖例中明確定義
- 若跨多張圖使用同一群組，顏色應延續一致
- 圖表中同一系列資料需維持相同色彩
- 流程圖、架構圖與統計圖若屬同一專案群組，可共用色彩語義

### 6. 圖例規範
- 圖例固定右下角
- 若圖例不必要則可省略
- 圖例尺寸要小，不可搶主畫面焦點
- 圖例顏色與符號要與主圖一致

### 7. 統計圖表專屬規範
- 數字圖表必須有標題、單位、資料來源或註記
- 若使用圓餅圖，類別建議不超過 5 個，超過則改用長條圖
- 若要比較大小，優先用長條圖，不優先用圓餅圖
- 若要看時間變化，優先用折線圖
- 若要呈現 Likert 問卷，優先用堆疊長條圖
- 若資料類別很多，優先用水平長條圖
- 若有排名，按數值排序
- 圖表不得 3D 化
- 禁止使用過度裝飾陰影或立體效果
- 軸標籤需清楚、簡短、對齊
- 百分比圖需標示總體基數 n 值（若有）
- 若顯示平均數、標準差或信賴區間，需清楚標註
- 若同一頁有多張圖表，色彩語義要一致

## 五、字型與語言規範

### 1. 預設語言
- 一律使用繁體中文
- 文字措辭優先採臺灣常用用語

### 2. 預設字型
- 首選：標楷體風格
- 若系統存在 DFKai-SB / BiauKai / 標楷體相容字型，優先使用該字型
- 若系統沒有可用的標楷體相容字型，才可尋找合法可免費使用的繁體中文替代字型
- 替代字型需具明確開源或免費授權，且能正常顯示繁體中文
- 禁止使用來源不明、授權不明、禁止再散布的字型

## 六、PPTX 版面優先原則

若用途為簡報，請將最終成果視為 PPTX 頁面設計，而不是單純圖片渲染。

### 優先使用的 PPTX 元件
- 文字方塊
- 圓角矩形
- 線條與箭頭
- 表格
- 圖示
- 色塊背景
- 分區標題
- 註解框
- 圖片
- 圖表元件

### 不建議依賴
- 過度複雜的 SmartArt 本體
- 難以後續修改的群組物件
- 純圖片式主版面
- 3D 圖表
- 過度裝飾的預設圖表樣式

---

## 七、圖檔裁邊與輸出規範

### 1. PNG 匯出前後處理
- 匯出 PNG 後，進入 Word 前必須先自動裁掉四周多餘空白
- 以非透明像素或非背景色範圍為主要內容邊界
- 預設保留極小安全邊界，避免裁切到外框
- 不可直接把大量白邊圖片嵌入 Word

### 2. SVG / PNG 輸出要求
- 背景需可控，若文件版面允許，優先保留透明背景
- 圖片解析度需足夠印刷或投影使用
- 內容不得因裁邊而破壞圖例、標題、註解

---

## 八、Word 文件整合與交互參考規範

若產出 DOCX，必須嚴格遵守以下規則：

### 1. 圖表順序
- 依文件內容出現順序插入
- 圖號必須連續，不可跳號
- 新增或刪除圖表後，應重新排序與更新交互參考

### 2. 圖表標題
- 每張圖都必須有圖說標題
- 標題格式統一，例如：
  - 圖 1　概念審查流程圖
  - 圖 2　平台整體架構圖
  - 圖 3　滿意度長條圖
- 圖說位置固定，建議置於圖下方
- 文字樣式、字級與對齊方式需一致

### 3. 內文引用
- 內文必須引用每一張圖
- 引用格式統一，例如：
  - 如圖 1 所示
  - 詳如圖 2
  - 問卷結果如圖 3 所示
- 不可出現文件中有圖但正文完全未提及的情況

### 4. 交互參考
- Word 內文中的「圖 1」「圖 2」等必須使用交互參考機制
- 不可只寫死文字號碼
- 更新欄位後，圖號需能自動同步

---

## 九、檔案輸出規範

每次產圖至少輸出以下檔案：

### 必要輸出
1. 原始文字描述檔或資料檔
2. 可視化輸出檔（SVG 或 PNG）
3. PPTX 可編輯版

### 視需求輸出
4. Word 嵌圖版
5. draw.io XML（若專案另有相容需求）
6. CSV 或 JSON 原始數據檔（若是統計圖表）

### 建議副檔名
- Mermaid：`.mmd`
- PlantUML：`.puml`
- Graphviz：`.dot`
- 結構化版面描述：`.json`
- 數據檔：`.csv`
- 圖片：`.png` / `.svg`
- 簡報：`.pptx`
- 報告：`.docx`

## 十、自動檢查規則

每次產圖完成後，必須自動檢查以下事項：

### 結構檢查
- 是否已正確選擇圖種
- 是否使用對應文字表示方式
- 是否存在過多交叉線
- 是否同層節點數超過 4 個
- 是否節點文字超過 12 字
- 是否流程方向混亂
- 若為統計圖表，是否選用合適圖種

### 美學檢查
- 是否標題置頂置中
- 是否同層節點尺寸一致
- 是否顏色超過 3 類
- 是否箭頭樣式一致
- 是否圖例位於右下角
- 是否有足夠留白
- 是否存在過密區塊
- 是否相同顏色代表相同或相似群組
- 若為圖表，是否符合數值表達原則

### Word 檢查
- 是否產出原始描述檔
- 是否產出 PNG / SVG
- 是否產出 PPTX
- 若為報告用途，是否同步產出 Word 嵌圖版本
- PNG 是否已裁邊
- 所有圖說是否存在
- 圖號是否正確排序
- 內文是否都有交互參考

## 十一、Codex 執行流程

每次接到畫圖需求時，依以下順序處理：

### Step 1：解析需求
先判斷：
- 用途是簡報、報告、技術文件還是內部草圖
- 圖的類型為何
- 是否需要可編輯 PPTX
- 是否需要 Word 嵌圖
- 是否屬於數值型資料視覺化

### Step 2：選擇圖種表示法
依圖類型自動切換：
- 流程圖 → Mermaid flowchart
- 系統架構圖 → PlantUML / Mermaid
- 網路拓樸圖 → Graphviz DOT
- 階層圖 → Mermaid / JSON
- 時序圖 → PlantUML sequence
- 時程圖 → Mermaid gantt
- 提案概念圖 → JSON / Python 版面結構
- 統計圖表 → JSON / CSV / Python dict

### Step 3：決定字型方案
- 先檢查系統是否有標楷體相容字型
- 若有，直接使用
- 若沒有，尋找合法免費且可顯示繁體中文的替代字型
- 記錄實際使用字型、來源與授權

### Step 4：生成結構草稿或資料定義
先產生簡潔、可維護的文字描述檔或資料檔，不要一開始就硬編圖片。

### Step 5：套用美學模板
將結構轉為簡報版面時，強制套用：
- 標題置頂置中
- 同層節點統一尺寸
- 每列不超過 4 個節點
- 主流程左到右
- 輔助流程虛線
- 三色限制
- 相同顏色代表相同或相似群組
- 箭頭方向一致
- 圓角矩形
- 圖例右下角
- 若為圖表，使用合適的統計圖模板

### Step 6：匯出圖片並裁邊
- 匯出 PNG / SVG
- PNG 進入 Word 前先 trim 四周空白
- 檢查裁邊後內容完整

### Step 7：輸出成果
至少輸出：
- 原始描述檔或資料檔
- PNG / SVG
- PPTX

需要報告時，再輸出 DOCX，並加入圖說、交互參考與圖號順序檢查。

### Step 8：自動檢查
若發現：
- 節點過多
- 文字過長
- 顏色超標
- 排版擁擠
- 箭頭交叉過多
- 字型不符規範
- Word 圖號或引用不一致
- 圖表選型不合理

則自動回退修正後再輸出。
